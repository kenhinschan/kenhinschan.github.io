mysql索引失效 && 索引优化

- 索引是检索、与排序的数据结构

- 最左前缀原则，（带头大哥不能死）（1楼，2楼，3楼，4楼）没有1楼就上不了2楼

- 中间兄弟不能断，（1楼，2楼，3楼，4楼） ，同样没有了2楼就上不了3楼

- 范围条件之后 in  like %kkk  i>9

- 索引覆盖

- where 条件使用了不等于、null or 索引会失效 

- order by  最左前缀原则

- ground by 分组前先排序，最左前缀原则，尽量使用where条件替换having

explain 分析SQL语句

  • 使用 explain 关键字来查看执行计划，是否命中索引

  • 通过计算某列的区分度，来判断该列是否适合新建索引

    • id字段 id相同时，执行顺序由上至下，id数值大的，优先执行，最后是null

    • select_type字段： 查询类型 外层查询（主查询）、子查询，临时表查询

    • type字段 性能：ref > index > all （all性能最差）

    • key字段 索引实际是否被用到 ，null值索引失效

    • key_len 索引长度

    • ref字段 索引是否是常量

    • rows字段 行数越小越好

    • Extra字段 不能出现Using filesort, Using temporary

show profiles 数据库自带sql性能分析工具

  • Show profile 是mysql 提供可以用来分析当前会话中语句执行的资源消耗情况。可以用于sql 调优的测量

    • 1、先开启

      • 命令：show variables like ‘profiling%’;

      • 命令：set profiling=on;

    • 2、运行SQL

    • 3、 查看结果：Show profiles；

    • 4、日常开发需要注意的结论：

      • 4.1 converting HEAP to MyISAM ： 查询结果太大，内存都不够用了，往磁盘上搬了；

      • 4.2 creating tmp table ：创建临时表，拷贝数据到临时表，然后再删除；

      • 4.3 copying to tmp table on disk ：把内存中临时表复制到磁盘，危险！！！

      • 4.5 locked

        注：以上四个中若出现一个或多个，表示sql 语句 必须优化。

mysql 锁机制

  • 1、表锁（偏读） - MyIsAm 无死锁 低并发，

    • 读锁（共享锁）

    • 写锁 （排他锁、独占锁）

  • 2、行锁（偏写） - InnoDB

    • 索引失效是 行锁会升级为表锁

    • 间隙锁范围条件会阻塞 （A事务范围条件修改数据，事务还未提交）同时（B事务添加范围条件内不再存在的数据，会出现阻塞）

    • 脏读 - 脏读发生在一个事务A读取了被另一个事务B修改，但是还未提交的数据。假如B回退，则事务A读取的是无效的数据。

    • 不可重复读 -

    • 幻读 -

  • 查看数据库锁情况：show open tables 或者 show open tables from database;

  • 查看服务器状态：show status like ‘%lock%’

    • 如当Table_locks_waited与Table_locks_immediate的比值较大，
      则说明我们的表锁造成的阻塞比较严重，可能需要调整Query语句，
      或者更改存储引擎，亦或者需要调整业务逻辑。当然，具体改善方式必须根据实际场景来判断。

    • 而Innodb_row_lock_waits较大，则说明Innodb的行锁也比较严重，
      且影响了其他线程的正常处理。同样需要查找出原因并解决。
      造成Innodb行锁严重的原因可能是Query语句所利用的索引不够合理
      （Innodb行锁是基于索引来锁定的），造成间隙锁过大。也可能是系统本身处理能力有限，则需要从其他方面来考虑解决。

  • 查看服务器配置参数：show variables like ‘%timeout%’;
